## 3. 技术实现详情

### 3.1 本地模型部署（Ollama + Qwen2.5）
- 通过 Ollama 桌面版在本地拉取并加载 `qwen2.5:3b / 7b / 14b`，后端以 `ChatOllama` 连接 `http://localhost:11434`；
- 采用"Embedding 走 CPU、LLM 走 GPU（可选）"的分工，节省显存；
- 模型按请求参数 `model` 热切换，切换时重建 `AgentOrchestrator` 实例。

### 3.2 数据爬取
- `crawler/npc_crawler.py`：国家法律法规数据库二期 API（搜索 → 详情 → 下载 DOCX → python-docx 解析）；
- `crawler/cleaner.py`：清洗并仅保留宪法 + 法律，过滤地方法规 / 司法解释 / 案例；
- 产出 `data/raw/` 原始文本与 JSON，覆盖主要法律部门。

### 3.3 文本分段与向量化
- `rag/chunker.py`：长法条（>500 字）按项 / 句拆分并保留法条前缀；短法条（<150 字）扩写上下文；
- `rag/embeddings.py`：使用 `bge-large-zh-v1.5`（1024 维）编码，CPU 推理；
- 统计：8798 条原始分段 → 8835 条最终向量。

### 3.4 向量索引（FAISS）
- `rag/indexer.py`：构建 `IndexFlatIP`（归一化内积等价于余弦相似度）；
- 已生成 `data/vectors/npc.faiss` + `npc_metadata.json`；
- `rag/retriever.py`：向量检索 + 同法条折叠去重。

### 3.5 RAG 问答核心
检索侧：问题编码 → FAISS 余弦检索 top_k → 同法条折叠去重 → 拼装"法律名 + 条款编号 + 正文"上下文。
生成侧：统一 Prompt 模板（`QA` / `SUMMARY` / `RETRIEVAL`）驱动本地 LLM，输出带引用来源的答案。

```
用户问题 → 向量编码 → FAISS 检索(top_k) → 同法条折叠去重
         → 拼接结构化上下文 → Prompt 模板 → Ollama 生成 → 答案 + 引用来源
```

### 3.6 多 Agent 调度结构
基于 LangChain 1.x，三类角色 Agent（检索 / 问答 / 总结），三种协作模式：
- **串行（sequential）**：检索 Agent → 问答 Agent；
- **并行（parallel）**：LLM 拆子问题 → `ThreadPoolExecutor` 并发检索 → 问答 Agent 汇总（拆分失败降级串行）；
- **合作（hierarchical）**：调度 Agent（ReAct）自主决策调用子 Agent。

**Agent 模式选择策略（核心权衡）**

| 模式 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| sequential | 单一、明确的法条查询 | 可控、延迟低、消耗小 | 不擅长综合性 / 多子问题 |
| parallel | 综合性、可拆分的复杂问题 | 覆盖多视角、召回更全 | 子问题拆分不稳定、延迟与 token 更高 |
| hierarchical | 需总结 / 对比 / 多步推理 | 灵活、体现多 Agent 协作 | 决策不确定、可能调用错误工具 |

当前由用户在前端手动选择模式；系统尚未实现"按问题类型自动选模式"的策略。

### 3.7 API 服务
- `api/server.py` 提供 `/chat`、`/search`、`/health`、`/clear_session`；
- 启动预加载索引，自动生成 Swagger UI（`http://localhost:8000/docs`）。

## 4. 遇到的问题（已遇到并记录）

### 4.1 本地 AI 启动相关问题
- **服务冷启动慢**：Ollama 首次加载 `qwen2.5:7b/14b` 权重耗时长，且首条请求因模型未驻留显存而延迟显著；
- **推理超时与不稳定**：本地 LLM 在复杂 prompt 下易超过默认超时；已缩短 timeout（180s→60s）并启用 `num_retries=2` 自动重试；
- **资源与模型档位权衡**：`3b` 响应快但法条引用易失真，`14b` 准确但本地无 GPU 时 CPU 推理很慢；需在"速度 / 精度"间按演示场景取舍；
- **上下文长度限制**：长法条经分块后拼接，仍可能因上下文窗口受限被截断，需靠"同法条折叠"压缩；
- **模型切换成本**：切换 `model` 会重建 `AgentOrchestrator`（重新连接 Ollama、预加载检索器），高并发下偶有抖动。

### 4.2 Agent 策略相关问题
- **并行模式子问题拆分不稳定**：依赖 LLM 拆问，偶发只返回 1 个子问题，触发降级为串行，影响并行收益；
- **合作模式决策不确定**：ReAct 调度 Agent 可能选错子工具（如本应检索却直接总结），结果不可完全预期；
- **缺乏自动选模式策略**：目前靠人工在前端选 sequential / parallel / hierarchical，未根据问题类型自动路由；
- **多 Agent 调用链耗时叠加**：合作 / 并行模式多次调用 LLM，端到端延迟明显高于串行。

### 4.3 其他工程问题
- **爬虫限流**：连续请求约 20 次后会被限流，需控制间隔与 pageSize；
- **长法条上下文溢出**：已通过分块 + 保留前缀缓解；
- **短法条语义不足**：已通过分块扩写增强召回。

### 5.2 技术待解决（深度优化）
- **本地 AI 启动与推理加速**：
  - 引入模型预热 / 常驻机制，降低冷启动延迟；
  - 评估 Ollama 量化（如 q4_K_M）以在 CPU 环境提速；
  - 探索 GPU 显存不足时的显存—内存协同与批处理；
- **Agent 模式自动选择策略**：
  - 设计问题分类器（规则 / 轻量模型），按"单一 / 综合 / 需总结"自动路由到对应模式；
  - 为合作模式增加工具调用约束与回退，降低错误工具调用概率；
  - 引入超时与降级统一治理，保障演示稳定性；
- **检索与生成质量**：
  - 引入混合检索（向量 + 关键词 BM25）与重排序（rerank）；
  - 长上下文管理：对超长法条采用层级检索 / 摘要前置；
- **评测体系**：建立准确率、引用准确率、相关性等量化指标与基线对比。
