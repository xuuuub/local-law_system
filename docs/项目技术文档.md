# 基于本地大语言模型的法律法规智能问答系统
## 项目技术文档

> 版本：v1.0
> 技术栈：Python · Ollama · LangChain 1.x · FAISS · FastAPI · Streamlit
> 部署形态：完全本地化（数据 / 向量索引 / 大模型均离线运行）

---

## 1. 项目背景与目的

在大模型快速发展的背景下，法律领域对智能问答与信息抽取提出了更高要求。面对海量法律法规文档，如何构建自主知识库并结合本地大语言模型（LLM）进行问答，是重要的工程实践方向。

本项目以"国家法律法规数据库（flk.npc.gov.cn）""北大法宝（pkulaw.com）"等公共法律数据源为基础，通过网页爬虫下载法律文本，构建本地法规知识库，并利用 RAG（检索增强生成）+ 多 Agent 技术，构建支持多轮交互、任务调用的智能问答系统。

核心目标：
- 全程离线：数据、向量索引、大模型均在本地运行，无云端依赖；
- 可解释：回答附带法条引用来源；
- 可扩展：支持多 Agent 协作调度与多种问答模式。

---

## 2. 技术选型

| 模块 | 技术 | 说明 |
|------|------|------|
| 本地 LLM | Ollama + Qwen2.5（3b / 7b / 14b） | 本地推理，模型可热切换 |
| 文档爬取 | Python（requests + BeautifulSoup / 官方 API） | 国家法律法规数据库、北大法宝 |
| 文本分段 | `rag/chunker.py` | 长法条拆分、短法条扩写 |
| 向量化 | `sentence-transformers` / BAAI/bge-large-zh-v1.5（1024 维） | CPU 推理，省 GPU 给 LLM |
| 检索 | FAISS（IndexFlatIP） | 余弦相似度检索 |
| Agent 框架 | LangChain 1.x（LangGraph Agent） | 多 Agent 协作 |
| 系统接口 | FastAPI | RESTful API + Swagger UI |
| 前端 | Streamlit | 可视化问答界面 |

---

## 3. 系统总体架构

系统采用四层架构：数据层 → 检索层 → 生成层 → 调度层，前端通过 API 与后端交互。

```
用户问题
    │
    ▼
┌─ Agent 调度层（LangChain）────────────────────────┐
│  sequential：   串行（检索 Agent → 问答 Agent）    │
│  parallel：     并行（拆子问题 → 多路并发检索）     │
│  hierarchical： 合作（调度 Agent + 3 子 Agent）     │
└──────────────────────┬───────────────────────────┘
                       │
    ┌──────────────────┼──────────────────┐
    ▼                  ▼                  ▼
检索 Agent         问答 Agent          总结 Agent
(FAISS 搜索)       (法条 → 答案)       (法条 → 摘要)
    │                  │                  │
    └──────────────────┴──────────────────┘
                       │
                       ▼
              带引用来源的最终回答
                       │
            ┌──────────┴──────────┐
            ▼                     ▼
        FastAPI 后端          Streamlit 前端
```

## 4. 模块化设计与实现

### 4.1 数据爬取模块（`crawler/`）

| 文件 | 职责 |
|------|------|
| `npc_crawler.py` | 国家法律法规数据库爬虫（二期 API：搜索 → 详情 → 下载 DOCX → python-docx 解析） |
| `pkulaw_crawler.py` | 北大法宝爬虫 |
| `cleaner.py` | 文本清洗、分段、法律类型过滤（仅保留宪法 + 法律，过滤地方法规 / 司法解释 / 案例） |
| `utils.py` | 日志、JSON 存储等公共工具 |

产出：`data/raw/` 下 166 个 txt + 8 个 json 原始数据，覆盖宪法、民法典、刑法、三大诉讼法、劳动法、行政法、公司法、知识产权、经济法等。

### 4.2 文本分块与向量化（`rag/`）

- `chunker.py`：长法条（>500 字）按项 / 句拆分并保留法条前缀；短法条（<150 字）扩写上下文增强语义。
- `embeddings.py`：加载 `bge-large-zh-v1.5`（1024 维），归一化向量，CPU 推理。
- 统计：84 部核心法律，8798 条原始分段，8835 条最终向量。

### 4.3 向量索引与 RAG 核心（`rag/`）

- `indexer.py`：`FaissIndexer` 构建 `IndexFlatIP` 索引（向量归一化后内积等价于余弦相似度），`save/load` 索引与元数据。索引文件：`data/vectors/npc.faiss` + `npc_metadata.json`。
- `retriever.py`：向量检索 + 同法条折叠去重，避免重复法条浪费上下文。
- `prompts.py`：统一管理的 Prompt 模板（检索 / 问答 / 总结），Agent 与 RAG 共用。

RAG 主链路：用户问题 → FAISS 检索 → 拼接上下文 → Prompt → Ollama 生成 → 答案 + 引用来源。

### 4.4 多 Agent 调度（`agents/`）

基于 LangChain 1.x，实现三套协作模式（`agents/orchestrator.py`）：

| 模式 | 说明 | 对应需求 |
|------|------|----------|
| `sequential` | 检索 Agent（ReAct + law_search 工具）→ 问答 Agent（LCEL chain） | 基础 Agent 组合 |
| `parallel` | LLM 拆 2~3 子问题 → `ThreadPoolExecutor` 并发检索 → 问答 Agent 汇总 | 并行模式 |
| `hierarchical` | 调度 Agent 把检索 / 问答 / 总结包装成工具，自主决策调用 | 多 Agent 合作 |

- 三个角色 Agent：检索 Agent、问答 Agent、总结 Agent；
- 检索 / 调度 Agent 用 LangGraph ReAct，问答 / 总结 Agent 用 LCEL chain；
- `chat()` 方法用 `session_id` 维持多轮对话上下文（最近 3~5 轮）。

### 4.5 API 服务（`api/server.py`）

FastAPI 服务，启动时预加载索引，后续请求秒级响应：

| 端点 | 方法 | 说明 |
|------|------|------|
| `/chat` | POST | 多 Agent 问答（支持 mode / model / session_id） |
| `/search` | POST | 纯向量检索（不调用 LLM） |
| `/health` | GET | 健康检查（索引大小、维度） |
| `/clear_session` | POST | 清空指定会话历史 |

自动生成 Swagger UI（`http://localhost:8000/docs`）。

### 4.6 可视化前端（`frontend/streamlit_app.py`）

Streamlit 界面，支持：
- 模型切换（qwen2.5 3b / 7b / 14b）；
- Agent 模式切换（串行 / 并行 / 合作）；
- 对话式交互、示例问题、系统状态检测；
- 通过 HTTP 调用后端 `/chat`，浏览器会话生成唯一 `session_id` 隔离对话历史。

## 5. 运行与部署

### 环境
- Python 3.13，可选 NVIDIA GPU
- 已安装 Ollama 桌面版

### 安装与启动
```bash
pip install -r requirements.txt -i https://pypi.tuna.tsinghua.edu.cn/simple
ollama pull qwen2.5:3b        # 或 7b / 14b
conda activate law

# 终端 1：API
python scripts/run_api.py
# 终端 2：前端
streamlit run frontend/streamlit_app.py
```
访问 `http://localhost:8501` 使用前端，`http://localhost:8000/docs` 测试 API。

### 更新法条
```bash
python scripts/run_crawler.py --source npc --type law --keywords 新法名 --max 10
python scripts/build_index.py
```

## 6. 测试与评估

- 构造结构化标注测试集（每条含 `question` / `expected_answer` / `category` 等字段），覆盖 10 个法律类别，满足"每组至少 100 条"的要求；
- `evaluation/run_eval.py`：批量评估脚本，自动填充 `retrieved_text` + `model_output`；
- `evaluation/auto_eval.py`：自动评分。

测试数据集格式示例：
```json
[
  {
    "question": "试用期最多可以约定多久？",
    "expected_answer": "依据《劳动合同法》第十九条，劳动合同试用期不得超过六个月。",
    "category": "劳动法"
  }
]
```

## 7. 总结与展望

本项目实现了"自爬文档 → 向量索引 → RAG → 多 Agent 调度 → API → 可视化前端"的完整链路，全程本地化运行，具备可解释（引用来源）与可扩展（多模式 Agent）的特点。

后续可优化方向：
- 检索策略增强（混合检索 / 重排序）；
- 更大规模法律语料与更细粒度评测；
- 前端交互与多模态展示优化。
